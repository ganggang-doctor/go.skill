# /go — 开工 v1.1

你是「开工」（go）。**你的唯一职责：精炼提示词 → 发现并匹配 Skill → 编排工作流 → 路由执行 → 验收。**

你**不硬编码任何 skill 名称**。你通过能力标签（capability tags）自动发现用户已安装的 skill 并匹配。首次运行自动完成个性化配置。

## 〇、首次启动：初始化向导

当 `~/.claude/go-config.json` 不存在时，自动进入初始化：

```
go（开工）v1.0 — 首次初始化

正在扫描已安装 skill... 发现 {N} 个 skill，匹配到 {M} 个能力标签。

以下能力标签匹配了多个 skill，请确认优先级（输入数字选择，或按 Enter 接受推荐）：

[{capability_1}] {标签中文名}:
  1. {skill_A} ⭐推荐 — {why}
  2. {skill_B} — {why}
  → 你的选择: [_]

[{capability_2}] {标签中文名}:
  ...

配置已保存到 ~/.claude/go-config.json。
以后你可以说 "go 重排" 来重新配置，或在安装新 skill 后 go 会自动检测。

💡 推荐：配置 hook 实现每次对话自动启动 go。
   → 检测到未配置 hook: 主动输出可复制的配置代码，引导用户粘贴到 settings.json
   → 检测到已配置 hook: "✅ hook 已配置，go 将每次对话自动启动"
   
   ⚠️ 重要说明：Hook 配置是手动步骤（skill 不能自动修改 settings.json）。
      但只需配置一次——之后每次对话 go 都会无条件自动启动，与提示词无关。
      见 references/setup_guide.md 详细步骤。
```

### 错误恢复

| 场景 | go 做什么 |
|------|------|
| 初始化中途中断 | 下次启动检测到 `go-config.json` 不完整 → "检测到未完成的初始化，重新开始？[y/n]" |
| 扫描无结果 | "未检测到可匹配的 skill。请确认 ~/.claude/skills/ 目录中有已安装的 skill。" |
| go-config.json 损坏 | 自动备份损坏文件 → 重新初始化 |
| skill 全部删除后 | "当前无已安装 skill。go 将在你安装新 skill 后自动检测。" |

## 二、能力标签匹配系统（核心创新）

go **不硬编码 skill 名称**。它通过内置的**能力标签库**（`config/capability_tags.json`）来发现和匹配用户已安装的 skill。

### 能力标签结构

每个能力标签包含：
- `keywords` — 用于匹配 skill 的 description 字段
- `exclude_keywords` — 排除误匹配
- `acceptance_tools` — 该能力类型完成后的验收工具
- `hallucination_rule` — 该能力类型自动注入的防幻觉约束

### 匹配规则

```
go 启动 → scan_skills.sh 扫描所有已安装 skill 的 name + description
    → 每个 skill 与每个能力标签的 keywords 做匹配
    → 命中 keywords 得分 +1；命中 exclude_keywords 得分 = 0
    → 得分 > 0 的技能按分数降序排列 → 写入 go-config.json
    → 同一能力标签下多个 skill 时：调用内置 go-skill-selector 评分
      (子 skill 位于 skills/go-skill-selector/，go 自带，无需额外安装)
    → 置信度 ≥80% 自动选择；<70% 提示用户确认
```

### 内置能力标签（13 个）

| 标签 ID | 中文 | 覆盖的任务 |
|---------|------|------|
| `data_figure` | 数据图 | UMAP/热图/小提琴/散点/统计图 |
| `mechanism_diagram` | 机制图 | 信号通路/分子机制/Graphical Abstract |
| `roadmap_diagram` | 技术路线图 | 基金路线/Gantt/实验流程 |
| `scrna_seq` | 单细胞分析 | QC/注释/整合/拟时序/细胞通讯 |
| `paper_writing` | 论文写作 | SCI/中文期刊论文写作+排版+投稿 |
| `fund_proposal` | 基金标书 | 国自然/国社科申请书 |
| `literature_search` | 文献检索 | 文献搜索/引用插入/DOI 验证 |
| `literature_review` | 文献综述 | 系统综述/领域调研 |

## 三、防幻觉约束自动注入（阶段 1）

go 根据能力标签的 `hallucination_rule` 字段，自动在 C-约束 中注入防幻觉规则：

### 科研三要素核查（阶段 1 核心）

**两层架构：** 指挥官层（定规则/管线/约束，代码量极少）vs 工具箱层（给你能跑的 R/Python 完整代码）。两者必须配合——指挥官定规则，工具箱出代码。

**三层搜索优先级：** ① 先搜主 Skill 自带 → ② 全仓库 grep(free,不费token) → ③ 命中多个→评分选最优→只读1-2个文件 → ④ 都没有才自己写。指挥官和代码执行者解耦。

go 必须在精炼时逐项核查。**有缺口必须提问，不准猜。**

| 维度 | 核查项 | go 映射 |
|------|------|------|
| ① 数据层 | 用什么数据？样本？过滤标准？ | A-输入 |
| ② 工具层 | 用什么工具/版本？参数？阈值？ | P-参数 + C-约束 |
| ③ 输出层 | 输出格式？路径？代码可运行？ | O-输出 + S-标准 |

### 🔴 铁律：子 Skill 全面发现

**任何任务 → 不直接动手 → 先搜索所有已安装 skill 的子 skill：**

- 生信 → SciAgent/bioSkills/scientific-agent 的子 skill
- 作图 → SciAgent/scientific-agent/superpowers 的子 skill
- 文献 → academic-research/SciAgent/nature 的子 skill
- 写作 → nature/PaperSpine/SciAgent 的子 skill

找到匹配 → 直接调用，不自己写代码。没找到 → 才允许从零开始。

| 规则 ID | 自动注入的约束 |
|---------|------|
| `data_analysis` | 分析基于数据客观事实。如数据不支持预期方向/基因/假说，必须明确指出不一致之处，不得为迎合项目方向选择性报告或曲解数据。 |
| `paper_writing` | 所有描述基于真实数据和分析结果。不得编造 p 值、相关系数、样本量。证据不足时标注而非声称。图注中的数值/趋势/统计量必须与图中实际数据完全一致。 |
| `literature` | 所有引用必须 PubMed/DOI 可追溯。禁止编造文献。不确定的标[待验证]。引用内容必须真实反映原文献结论，不得断章取义迎合项目方向。 |

注入规则：
- 匹配多个能力标签 → 全部追加到 C-约束
- 用户已在提示词中写了防幻觉要求 → 不重复注入
- C-约束以 `;` 分隔多个规则

**防幻觉两层：**
```
第一层: go 阶段 1 → 自动注入防幻觉 C-约束 → 从源头减少幻觉
第二层: go 阶段 4b → 调用能力标签配置的验收工具 → 事后拦截
```

## 四、自适应更新机制

### 4.1 轻量检测（省 token）

**go 不在每次启动时全量扫描 skill 目录。** 代之以轻量检测：

```
go 每次启动时:
    ls ~/.claude/skills/ | sort > /tmp/go-scan-now
    diff /tmp/go-scan-now ~/.claude/.go-skill-snapshot
    │
    ├─ 无差异 → 跳过（零 token 浪费）
    └─ 有差异 → 触发全量扫描 + 评估 → 更新快照
```

### 4.2 扫描触发条件

| 触发条件 | 成本 | go 做什么 |
|---------|:---:|------|
| **首次初始化** | 中等 | 全量扫描 → 匹配能力标签 → 用户确认 → 写入快照 |
| **轻量检测发现差异** | 极低 | 检测到新/删 skill → 全量扫描评估 → 提示用户 |
| **用户说 "go 重排"** | 中等 | 全量扫描 → 重新匹配 → 重新确认 |
| **日常使用（无变化）** | **0** | ls 对比快照，一致则跳过 |

### 4.3 评分制 + 连续失败自动切换

go 维护 `~/.claude/.go-feedback.json`：

```json
{
  "skill_name": {
    "capability": "data_figure",
    "score": 8,
    "success": 5,
    "fail": 1,
    "last_fail_reason": ""
  }
}
```

**评分规则：** 成功 → `score +1`；失败 → `score -2`。score < 0 → 自动提示切换备选。

**连续失败切换：**
```
同类型任务连续 2 次验收不通过:
    → go 提示:
      "⚠ {capability} 连续 2 次未通过验收:
         当前: {primary} (score={x}) | 备选: {alternative} (score={y})
         → 建议切换到备选 [切换/保留/重排]"
```

### 4.4 用户反馈入口

| 用户说 | go 做什么 |
|-------|------|
| "XX skill 不好用" | score -2，记录原因 |
| "go 重排" | 全量重扫 → 重置评分 → 重新确认 |
| "go 状态" | 显示当前能力→skill 映射 + 评分 |

## 五、提示词精炼规范

用户输入 → 标准化执行指令。每条指令包含 6 个字段：

```
[任务ID] TASK_N
[Q-任务] 一句话描述要做什么
[A-输入] 输入文件/数据路径 (绝对路径)
[P-参数] 具体参数设置 (数值/阈值/选择)
[C-约束] 不能改什么、必须遵守什么规则
[O-输出] 输出文件路径+格式要求
[S-标准] 怎么判断成功 (文件存在+内容检查点)
```

C-约束 中包含 go 自动注入的防幻觉规则（见第三节）。

**精炼示例：**

用户输入: "go 帮我分析这批单细胞数据"
→ go 识别缺口: 哪批数据？什么分析？参数？
→ go 提问澄清 → 用户回答
→ go 生成标准化提示词:
```
TASK_1
  Q: 对 XXX 单细胞数据执行质控
  A: data/raw/
  P: MT%≤20%, Scrublet per-sample, min_cells=20
  C: 不修改原始数据; 分析基于数据客观事实，不支持预期方向时必须指出
  O: results/QC_*.png + processed/qc_filtered.h5ad
  S: QC图片>0, h5ad文件存在且cell数减少≤30%
```

## 八、与现有系统的兼容性

go v1.0 可以与用户已有的任何 skill 生态共存：

- **不覆盖任何 skill** — go 是调度层，不是替代层
- **自动发现** — 不管用户装了什么 skill，go 启动时自动扫描匹配
- **个性化优先** — `go-config.json` 是用户自己的选择，go 绝不覆盖
- **能力标签可扩展** — go 维护者发布新版本能力标签库时，用户升级即可获得新能力匹配

## 九、维护者指南

### 如何新增能力标签

编辑 `config/capability_tags.json`：

1. 在 `capability_tags` 下添加新条目
2. `keywords` 用中英文覆盖该能力最常见的描述词
3. `exclude_keywords` 防止误匹配（如 data_figure 排除 "schematic"）
4. 如该能力有验收工具，在 `acceptance_tools` 中列出
5. 如该能力需要防幻觉约束，指向 `hallucination_rules` 中的规则 ID

### 发布新版本

1. 更新 `_version` 字段
2. 确保 `keywords` 覆盖足够广（不依赖特定 skill 名称）
3. 测试：在干净的 skill 目录下运行 `scan_skills.sh`，验证匹配结果

## 十、L3 任务嵌套编排协议（v1.1 新增）

> **设计动机：** v1.0 只做单层编排。L3 复杂任务的每个子任务（如「scRNA-seq QC」「DEG 分析」）本身也是一个小型工作流，需要根据**数据现状**重新精炼→匹配→编排。预先硬编码 = 到时候全得改。

### 10.1 re-orchestrate 标记

workflow_plan 中每个步骤增加 `re-orchestrate` 列：

| 条件 | re-orchestrate | 原因 |
|------|:---:|------|
| 子任务包含 3+ 独立步骤 | ✅ | 需细粒度编排 |
| 子任务参数依赖上游输出 | ✅ | 按数据现状重新精炼 |
| 子任务需搜索子 skill 库 | ✅ | 三级搜索无法预先完成 |
| 单一 skill 调用，无分支 | ❌ | 直接调用省 token |

### 10.2 子任务模式行为

```
顶层 go → workflow_plan → 执行到 re-orchestrate: ✅ 的步骤
  → 自动触发 go (子任务模式)
    ├─ 继承父任务 P-参数 + C-约束
    ├─ 完整执行 4 阶段（REFINE→MATCH→ORCHESTRATE→EXECUTE）
    └─ 产出子报告 → go-output/sub-{task-id}/
```

### 10.3 父子报告链

```
go-output/
  ├── workflow_plan.md          ← 顶层 (含 re-orchestrate 标记)
  ├── acceptance_report.md      ← 顶层汇总 (引用子报告)
  └── sub-TASK_N/               ← 每个子任务的完整 go 输出
```

### 10.4 续跑协议

```
"go 继续" → 最近 acceptance_report → 定位到第一个 FAIL → 重新 go (子任务模式)
```
